## 1. Что это
Самостоятельный **модуль онлайн-бронирования номеров** для размещения на VPS (Ubuntu 24) и подключения к сайтам на **Tilda**. Извлечён из готового сайта `azov-resort`, но как отдельный продукт без лишнего (без блога/территории/отзывов/услуг/аналитики/VK).

**Бронь = заявка без аккаунта гостя.** Оплата депозита — офлайн (админ отмечает вручную). Язык — русский. Деньги — в **копейках** (рубли × 100).

---

## 2. Ключевые решения по интеграции (обсуждено с владельцем)
- **Объект = домен.** У каждого Объекта свой публичный **домен** (поле `domain` в админке, не в `.env`). Гость заходит на `https://<домен-объекта>` → корень `/` сразу отдаёт **список номеров этого объекта** (объект определяется по `Host`-заголовку запроса). Никакого общего списка объектов, голубого хедера и `/o/[slug]` — объект это внутренняя сущность, гость про неё не знает.
- **Домены работают сами (On-Demand TLS).** Пользователь вписывает домен в админке и направляет на него DNS (A-запись на IP VPS). Caddy ловит любой домен (`https://` блок) и выпускает сертификат «по запросу», спрашивая у приложения `/api/tls-check?domain=…` — `ok` только для доменов привязанных к активному объекту. Никакой правки конфигов на сервере под новый объект не нужно. Админка — на отдельном домене `ADMIN_DOMAIN` (обычный сертификат).
- **«Объект» (Object/PropertyObject в коде)** — группа номеров (по адресу/зданию), сущность **внутренняя**. Поля: `name` (внутреннее имя), `domain` (публичный домен), `address` (пометка), `isActive`, `sortOrder`. В шахматке админки объекты — сворачиваемые группы; сначала создаётся Объект, затем внутри него номера. Удаление Объекта удаляет его номера (cascade).
- **Мульти-сайт.** Несколько сайтов Tilda на разных доменах → одна админка и БД. Каждый Tilda-сайт вешает кнопку (напр. «Номера») со ссылкой на домен «своего» Объекта.
- **Маршруты витрины (решено).** `/` — список номеров объекта, определённого по домену запроса; `/rooms/[roomSlug]` — карточка номера + бронирование (slug номера уникален глобально, генерируется автоматически из названия). Обвязка минимальна: подвал из настроек + кнопки «Назад» (слева сверху) и «На главную» (справа) на списке и на карточке (`nav_back_url`/`nav_home_url`, ведут на Tilda; по умолчанию `/`).

## 3. Стек (наследуется от донора)
- **Next.js 14.2** (App Router), React 18, TypeScript, `output: 'standalone'`.
- **Prisma 5 + PostgreSQL 16**. Схема синхронизируется через `prisma db push` (миграций нет / по желанию добавим).
- **Tailwind CSS**. Иконки `lucide-react`, анимации `framer-motion`, календарь `react-day-picker@9`, формы `react-hook-form` + `zod`, изображения `sharp`.
- **Авторизация только админ/сотрудник**: собственный JWT (`jose`, HS256, cookie) + `bcryptjs` для хэшей. Пользовательских аккаунтов нет.
- Деплой: **Docker Compose** (postgres + app), **Caddy** на VPS для поддоменов + автоматический Let's Encrypt SSL (без certbot). Приложение слушает на порту **3015** хоста (`127.0.0.1:3015:3000`).

Выпилено из донора: `recharts`, `@tiptap/*`, `nodemailer`/email (почтовые уведомления не нужны), VK-уведомления, аналитика, блог/территория/отзывы/услуги, `swiper` (не используется; мёртвые CSS-правила из globals.css удалены).

## 5. Модель данных (целевая, `prisma/schema.prisma`)
**Новое/изменённое относительно донора:**
- **`PropertyObject` (Объект)** — `id, name, domain? (@unique), address?, sortOrder, isActive, createdAt, updatedAt`, `rooms Room[]`. Сущность внутренняя; публичная привязка — через `domain` (см. §2). **Убрано:** `slug`, `description`, `publicUrl`.
- **`Room`** — добавлено `objectId` + relation на PropertyObject (`onDelete: Cascade`). Номера **не захардкожены** — CRUD из админки. `slug` генерируется автоматически из названия (пользователь не задаёт). **Поля (решено):** name, slug(авто), кратко/полно описание, базовая цена + сезонные периоды, **`capacity` (одно число — количество мест)**, площадь, этаж, sortOrder, **свободный список удобств** (`amenities`), галерея, активность, блокировки дат. **Убрано:** seoTitle/seoDescription, 4 булевых флага удобств, **`baseCapacity`/`extraCapacity` (слиты в одно поле `capacity`)**.
- **`AdminUser`** — `id, login, passwordHash (bcrypt), role (ADMIN|STAFF), isActive, createdAt`. Главный админ может создавать/удалять сотрудников; сотрудник пароль менять не может. **Все пользователи (включая главного админа) живут в БД.** Стартовый супер-админ при первом логине «посевается» из env (`ADMIN_LOGIN`/`ADMIN_PASSWORD_HASH`) в таблицу — см. `lib/admin-seed.ts` (`ensureSeedAdmin`). Дальше главный админ меняет свои логин/пароль из «Настроек» через `/api/admin/account`. Пароли — только хэш. **Важно:** bcrypt-хэш в `.env` содержит `$` — чтобы docker-compose не интерпретировал их как переменные, секреты передаются через `env_file`, а не через `environment` в `docker-compose.yml`.
- **`Booking`** — как у донора (даты, гости, депозит, цена в копейках, `priceBreakdown`, `status`, `paymentStatus`/`paidAt`, отмена, `source`). Статус `BLOCKED` из `BookingStatus` **убран** (по ТЗ не используется).
- **`RoomPricePeriod`, `BlockedDate`** — без изменений.
- **`Setting`** — key/value для настроек (§7).

**Удалено из донора:** `BlogPost, TerritoryEntry, Review, Service, PageView, ConversionEvent` и связанные enum.

## 6. Админка — три вкладки (по ТЗ)
1. **Бронирования** — шахматка. Слева столбец — **Объекты как сворачиваемые группы** (галочка раскрывает → номера объекта и брони по ним). Изменения против донора: статус показывать одной строкой «Статус» (без «Бронь:…, Оплата:…»); кнопка «**Удалить бронь**» (не «броню»); без статуса «Заблокировано».
2. **Номера** — CRUD объектов и номеров. Сначала создаётся **Объект**, затем номера внутри. Объект/номер можно редактировать/удалять (удаление объекта → удаляет его номера). Захардкоженных 8 номеров нет.
3. **Настройки** — см. §7.

## 7. Вкладка «Настройки» (состав — по ТЗ)
- **Документы для бронирования**: ссылки на документы, которые гость обязан отметить галочкой согласия перед созданием брони (без галочки бронь не создаётся).
- **Подвал**: реквизиты (организация/самозанятый), ссылки на те же документы.
- **Навигация карточки номера**: куда ведут кнопки «На главную» и «Назад» (ссылки настраиваются).
- **Контакты и адрес**.
- **Депозит**: тип (процент/фикс) и значение — для расчёта брони.
- **Время заезда/выезда** (check-in / check-out).
- **Управление доступом**: смена логина/пароля главного админа; добавление/удаление сотрудников (главный админ задаёт сотруднику пароль; сотрудник сменить его не может). Пароли — только хэш (bcrypt).

## 9. Подводные камни (наследие донора)
- **Деньги — в копейках** (× 100) — частый источник ошибок.
- **Менял `schema.prisma` → на деплое нужен `prisma db push`.** (Доработки v5 удалили колонки `slug/description/publicUrl` у объекта и `baseCapacity/extraCapacity` у номера — `db push` их отбросит, это ожидаемо.)
- **Объект определяется по `Host`.** Корень `/` читает `headers().get('host')` → `normalizeDomain` → ищет `PropertyObject.domain`. Локально/без домена объект не найдётся (покажется «нет объекта»). Для теста — задать домену объекта `localhost` или ходить через реальный домен.
- **On-Demand TLS.** Caddy выпустит сертификат только если `/api/tls-check?domain=…` вернул `ok` (домен привязан к активному объекту). Сначала объект+домен в админке, потом DNS.
- **`force-dynamic`** в layout'ах — БД недоступна при билде; публичные страницы дёргают prisma напрямую. Не снимать без защиты вызовов от билд-времени.
- **Баг календаря (фикс при порте):** в `BookingForm` дизейблилось `{ before: yesterday }` → вчерашний день оставался выбираемым. Чинить на `{ before: startOfDay(today) }` (+ `fromDate={today}`).
- **`amenities`** бывает массивом строк ИЛИ объектом-флагами (наследие донора) — нормализовать через `normalizeAmenities()` из `lib/utils`.
- **bcrypt-хэш в `.env`** содержит `$` → docker-compose интерпретирует их как переменные и обрезает значение. Фикс: секреты (`ADMIN_PASSWORD_HASH`, `ADMIN_JWT_SECRET` и др.) передаются через `env_file: - .env` в `docker-compose.yml`, а не через `environment`.
- **Картинки (`/uploads/[...path]`) и sharp.** Каждый запрос идёт с `?fm=webp` → трансформация через `sharp`. В Next standalone нативный `sharp` 0.33 (`@img/sharp-libvips-*`) **трейсится ненадёжно** → если модуль не грузится, отдаётся 500 → битые превью. Фиксы: (1) sharp в роуте подгружается **лениво** и при ошибке роут отдаёт **оригинал** (не 500); (2) Dockerfile **явно копирует** `node_modules/sharp` и `node_modules/@img` в runner. Симптом в логах: `[uploads] sharp недоступен…`.
- **Slug номера — авто-ASCII, не из названия.** Названия русские; кириллица в URL percent-кодируется и ломала открытие карточки (404). Slug генерируется как `room-<hex>` (`api/admin/rooms` POST), при редактировании **не меняется**. Старые кириллические slug чинит одноразовая миграция `scripts/fix-room-slugs.mjs` (запуск в `docker-entrypoint.sh` после `db push`).
